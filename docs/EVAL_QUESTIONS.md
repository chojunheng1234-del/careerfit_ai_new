# EVAL_QUESTIONS.md — CareerFit AI 자기 평가 질문

## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.

## 프로젝트 전체

- 이 서비스가 해결하는 문제는 무엇인가?
  → 취업 준비생이 자신의 전공과 역량이 어떤 직무와 맞는지 판단하기 어렵고, 많은 채용 공고를 직접 비교하기 힘든 문제를 해결한다. 사용자의 역량을 실제 채용 공고와 비교해 맞춤형 준비 방향을 제안한다.

- RAG를 사용하지 않으면 어떤 문제가 생기는가?
  → LLM이 학습된 정보만으로 답변해 실제 채용 공고와 다른 내용을 생성(Hallucination)할 수 있다. RAG를 사용하면 실제 공고를 근거로 답변하고 출처도 함께 제공할 수 있다.

- 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?
  → ChromaDB 검색 결과를 Gemini와 연결해 answer와 sources를 함께 반환하는 RAG 구현이 가장 어려웠다. 검색 결과를 프롬프트에 포함하고 응답 구조를 수정해 근거 기반 답변이 나오도록 해결했다.

## 데이터 관련

- 왜 이 데이터를 선택했는가?
  → 사용자의 전공과 스킬을 실제 채용 공고와 비교해 맞춤형 조언을 하기 위해 실제 채용 공고 데이터를 선택했다. 이를 통해 근거 있는 답변과 출처를 함께 제공할 수 있다.

- 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?
  → 현재는 교육용 목업 데이터와 일부 공고를 사용했기 때문에 최신 채용 정보가 아닐 수 있다. 실제 서비스에서는 최신 공고를 지속적으로 수집하고 검증하는 과정이 필요하다.

- SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?
  → SQLite는 조건에 맞는 데이터를 정확하게 조회하는 데 사용하고, ChromaDB는 의미가 비슷한 공고를 검색하는 RAG용으로 사용한다. 두 데이터베이스의 역할이 달라 함께 사용했다.

- 결측치와 중복을 어떻게 처리했는가?
  → Pandas를 사용해 필수 정보가 없는 데이터는 제거하고, 회사명과 직무명이 같은 공고는 중복으로 판단해 제거했다. 또한 스킬 표기를 통일해 검색 품질을 높였다.

## RAG 관련

- RAG 구조에서 ChromaDB는 어떤 역할을 하는가?
  → 사용자의 질문과 의미가 비슷한 채용 공고를 벡터 검색으로 찾아 Gemini에 전달하는 역할을 한다. 이를 통해 실제 공고를 근거로 답변할 수 있다.

- n_results 값을 높이면 어떤 트레이드오프가 생기는가?
  → 더 많은 근거를 사용할 수 있지만 관련성이 낮은 문서까지 포함되어 답변 품질이 떨어질 수 있다. 반대로 너무 적으면 근거가 부족해질 수 있다.

- 검색 결과 품질을 어떻게 검증했는가?
  → 검색된 공고가 질문과 관련 있는지 확인하고, answer가 실제 sources를 근거로 작성되었는지 비교해 검증했다.

- Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?
  → Hallucination은 LLM이 사실이 아닌 내용을 실제처럼 생성하는 현상이다. RAG는 실제 검색된 데이터를 함께 제공해 근거 기반으로 답변하도록 만들어 이를 줄여준다.

## API 관련

- /analyze 엔드포인트가 POST를 사용하는 이유는 무엇인가?
  → 전공, 스킬, 관심 직무처럼 여러 데이터를 JSON 형태로 전달해야 하므로 POST를 사용했다. GET보다 구조화된 데이터를 안전하게 전달하기에 적합하다.

- 응답 JSON에 sources를 포함시킨 이유는 무엇인가?
  → 어떤 채용 공고를 근거로 답변했는지 사용자에게 보여주기 위해서다. 답변의 신뢰성과 설명 가능성을 높일 수 있다.

- /health 엔드포인트가 왜 필요한가?
  → 서버가 정상적으로 실행 중인지 빠르게 확인하기 위해 사용한다. Docker와 Render의 Health Check에서도 활용할 수 있다.

## 보안 관련

- .env 파일을 GitHub에 올리면 어떤 일이 생기는가?
  → API Key가 외부에 노출되어 악용될 수 있다. 키가 비활성화되거나 과금 및 보안 문제가 발생할 수 있다.

- .env와 .env.example의 차이는 무엇인가?
  → .env는 실제 API Key 등 민감한 정보를 저장하는 파일이고, .env.example은 필요한 환경변수 이름만 안내하는 예시 파일이라 GitHub에 올려도 된다.

---

## Docker 관련

- Docker를 사용하는 이유는 무엇인가?
  → 개발 환경과 실행 환경을 동일하게 만들어 어디서 실행해도 같은 결과가 나오도록 하기 위해 사용한다. 배포도 더 쉽고 안정적으로 할 수 있다.

- Dockerfile에서 EXPOSE와 CMD의 역할은 무엇인가?
  → EXPOSE는 컨테이너가 사용할 포트를 문서화하는 역할을 하고, CMD는 컨테이너가 시작될 때 실행할 기본 명령어를 지정한다.
